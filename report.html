ABSTRACT

摘 要 Abstract

本研究設計並實作一套基於遷移學習的人臉表情品質自動評估系統,協助使用者從相簿中挑出表情崩壞的廢片(閉眼、嘴歪、模糊),並整合人臉偵測創新與嚴謹評估流程。

在數位攝影時代,每人每天產出大量照片,其中夾雜許多廢片(閉眼、嘴歪、模糊),手動逐張篩選耗時冗長。既有相簿軟體僅能做品質粗篩(模糊、過曝偵測),無法判斷「表情品質」這個高階語意。本研究以遷移學習為核心,在 1,060 張自建資料集上,自訓 MobileNetV3-Large(含 Squeeze-and-Excitation Attention)神經網路,並開發 Streamlit Web UI 讓使用者本機運行、完全離線地批次清理相簿。

在資料工程上,我們提出「三層 Fallback」人臉偵測策略:整合 MediaPipe short-range model、full-range model,以及自行設計的 letterbox 自動裁切機制(處理 iPhone 螢幕截圖的純色邊框),將漏失率從 27 張大幅降至 3 張(改善 88.9%)。在訓練流程上,採用兩階段微調(凍結特徵 → 解凍最後 3 個 inverted residual blocks),搭配 Weight Decay、Label Smoothing、RandomResizedCrop、RandomErasing 等正則化與資料增強策略,並透過 Test-Time Augmentation 與決策閾值自動掃描優化推論。

在嚴謹評估上,採 70 / 15 / 15 train / val / test 三段切分,於獨立 test set(n=193)達 75.1% accuracy,Bad 類 recall 0.828(廢片偵測成功率高),證明系統具備實際使用價值。我們也對比 MobileNetV2 與 V3-Large,驗證 SE Attention 對人臉細粒度任務的價值(+4 個百分點)。

研究貢獻 (1) 提出三層 fallback 人臉偵測架構,解決 iPhone 截圖等 edge cases; (2) 完整呈現遷移學習 + 兩階段微調流程,並以對照實驗驗證架構選型; (3) 整合端到端 Web UI,具備軟刪除、還原、視覺化等實際可用功能。

CHAPTER 01

01 / BACKGROUND 緣起與問題定義

我們從一個日常的痛點出發:相簿太亂、廢片太多、不敢隨意刪除。既有工具卻只能幫使用者挑出模糊或過曝的照片,無法理解「人臉表情品質」這個更貼近真正在意的層次。

1.1數位攝影時代的廢片困境

智慧型手機與連拍功能普及,使每位使用者的相簿快速膨脹到數千甚至上萬張照片。其中夾雜大量重複、品質不佳、或表情崩壞的廢片。儘管儲存空間日漸便宜,但「相簿太亂、找不到想要的照片」仍是現代人普遍的數位體驗痛點之一。

我們認為,問題不在「存不下」,而在「從上千張照片中找出明顯不會用到的那些」這個操作的時間與心理成本太高 — 連拍 10 張可能有 9 張是閉眼或表情怪,但要逐張點開、檢查、刪除,動輒耗時數十分鐘。

1.2既有解決方案的局限

我們盤點目前市面上的相簿整理工具:

產品提供的「自動篩選」能力核心局限
iCloud / Google Photos智慧分類、面孔識別、相似照片群組不會主動建議刪除,使用者仍需逐張決定
Gemini Photos (Mac)找出重複、模糊、相似照片僅基於像素相似度與品質指標,不理解表情
CleanMyMac / Photo Sweeper大量、模糊、低解析度偵測同上,無法分辨「光線好但表情崩壞」的照片
Apple「最佳照片建議」連拍中推薦最佳一張黑盒、不可調、無法批次處理

上述工具均聚焦於「低階品質指標」(模糊、過曝、解析度、重複),但對「人臉表情品質」這個高階語意層面束手無策。一張光線完美、構圖無懈可擊的照片,如果主角閉眼了,仍然是廢片。

1.3問題定義

PROBLEM STATEMENT 如何在「小資料集」與「完全本機運行」的條件下,訓練一個深度學習模型,對使用者相簿中的人臉照片自動判別「表情自然」與「表情崩壞」,並透過直覺的 Web UI 協助批次清理?

這是一個二元細粒度影像分類任務,具有三個關鍵約束:

  • 小資料集:學期專題時間有限,我們能蒐集到的有標籤資料僅約 1,000 張。
  • 本機運行:照片屬個人私密資料,使用者期待運算完全在本機完成,不上傳雲端。
  • 多元場景:使用者相簿包含自拍、合照、影片截圖、社群媒體下載圖等,模型必須對多種來源穩定泛化。

1.4專案目標

技術目標

建立完整的遷移學習管線,在小資料集條件下達到可實用的分類準確率,並以對照實驗驗證架構選型。

產品目標

實作可實際運行的 Web UI,具備批次掃描、視覺化結果、軟刪除與還原等完整功能,而非僅止於 Notebook demo。

學習目標

體現「設計思考 × 人工智慧」的學習目標 — 從使用者痛點出發、迭代原型、最後產出真正能解決問題的工具。

2.1關鍵設計原則

我們在規劃此系統時,將以下四個原則納入設計考量,並讓每個原則對應到一個具體可驗證的功能實作:

設計原則對應的使用者場景系統實作對應
隱私優先 使用者擔心私密照片上傳雲端 所有運算在本機完成(PyTorch + MediaPipe),無任何網路請求

2.2產品迭代

在開發過程中,我們經歷數次原型迭代。每一次迭代都針對前一版本的具體問題提出改善:

  • 初版(傳統 OpenCV):使用 Laplacian variance 偵測模糊、histogram 偵測過曝。結論:能抓低階品質問題,但無法判斷表情,確認此任務需要深度學習介入。
  • 第二版(PyTorch baseline):套用 MobileNetV2 + 凍結特徵層的兩階段訓練,val_acc 約 70-72%。確立遷移學習的可行性,但發現人臉偵測有許多漏失案例。
  • 第三版(三層 Fallback):新增 MediaPipe 雙模型 + letterbox 偵測,將人臉偵測漏失率從 27 張降到 3 張(改善 88.9%)。
  • 最終版(本報告):升級為 MobileNetV3-Large + SE Attention,並導入 TTA、決策閾值掃描、嚴謹三段切分評估等優化,test_acc 達 75.1%。

2.3從失敗案例驅動設計

本研究最關鍵的設計創新 — 三層 Fallback 偵測 — 並非來自理論推導,而是來自對失敗案例的詳細觀察。當我們發現單一模型漏掉 27 張照片時,我們逐張分析這些案例,發現它們可以歸類為三種失敗模式(肖像照、iPhone 截圖、極端 edge cases),進而設計出對應的三層降階策略(詳見第 5 章)。

設計反思 這個過程體現了設計思考中「從具體案例反推設計」的精神 — 與其先設計通用方案,不如先觀察真實失敗,再針對性提出對策。耐心檢視 edge cases,往往帶來最大的工程突破。

CHAPTER 03

03 / LITERATURE & TECH STACK 文獻回顧與技術選型

本章說明本研究的核心技術與文獻依據:遷移學習、MobileNetV3-Large、SE 注意力機制、MediaPipe 人臉偵測,以及 PyTorch + FastAPI + Vue 3 的部署選擇。

3.1遷移學習(Transfer Learning)

遷移學習的核心精神是「將在大型資料集(如 ImageNet 的 1,400 萬張圖)上預訓的網路,作為新任務的起點,而非從零開始訓練」。Yosinski et al. (2014) 在《How transferable are features in deep neural networks?》中實證:卷積網路前幾層學到的「邊緣、紋理、形狀」等通用特徵,可廣泛遷移至各種視覺任務。

對本研究而言,遷移學習是必要的選擇:

  • 資料集僅 1,060 張,從零訓練 MobileNetV3 必然嚴重過擬合。
  • ImageNet 預訓權重已學會「人臉、眼睛、嘴巴」等低階特徵,我們只需教模型「在這些特徵上,什麼組合 = 崩壞表情」。

3.2MobileNetV3-Large 與 SE Attention

MobileNet 系列由 Google 為行動裝置設計,核心目標是「在精度與運算量間取得平衡」。其演進路線:

  • MobileNetV1 (2017):提出 Depthwise Separable Convolution,大幅降低運算量。
  • MobileNetV2 (2018):引入 Inverted Residual + Linear Bottleneck,提升特徵表達能力。
  • MobileNetV3 (2019, Howard et al.):結合 NAS 自動搜尋最佳架構,並導入 SE Attention 模組與 Hard-Swish 激活函數。

SE Attention(Hu et al., 2018)是關鍵的注意力機制 — 它讓網路學會「哪些通道(channel)重要、哪些不重要」,並用 sigmoid 動態調整通道權重:

SE Attention 的三階段:
  # 1. Squeeze: 把每個通道壓成單一數值(global average pooling)
  z = GlobalAvgPool(x)                # (B, C, H, W) → (B, C)

  # 2. Excitation: 用 2 層 FC 學「通道重要性」
  s = sigmoid(W2 · ReLU(W1 · z))      # (B, C) → 通道權重 ∈ [0, 1]

  # 3. Scale: 用權重重新縮放原始特徵圖
  output = x ⊙ s                       # 通道乘法

對人臉表情這種細粒度任務,SE Attention 特別有效 — 它幫助網路自動聚焦於「眼睛、嘴角」等真正承載表情資訊的通道,而忽略背景、頭髮等非關鍵特徵。我們在第 6 章透過 V2 vs V3-Large 對照實驗驗證此論點。

3.3MediaPipe Face Detection

人臉偵測是本系統的前置步驟。我們選用 Google 開源的 MediaPipe(Bazarevsky et al., 2019, BlazeFace),原因:

  • 輕量高速:BlazeFace 在 CPU 上單張推論時間極短,適合即時批次處理。
  • 提供兩種模型:short-range(< 2 公尺,擅長肖像)與 full-range(< 5 公尺,擅長合照中遠距人臉),我們將兩者整合為 fallback 鏈。
  • 輸出 bounding box + 關鍵點:除了 bbox,還回傳眼睛、嘴巴等關鍵點(本研究僅使用 bbox)。

4.3Checkpoint 自描述設計

我們將模型 checkpoint 設計為「自描述」格式 — 不僅儲存權重,還包含後續推論所需的所有 metadata: 

# models/mobilenet_face.pth 的內容結構
checkpoint = {
    "state_dict": model.state_dict(),       # 訓練好的權重
    "arch":       "mobilenet_v3_large",    # 架構名
    "classes":    ["Bad", "Good"],         # 類別順序
    "img_size":   224,                       # 輸入尺寸
    "epoch":      9,                         # 最佳 epoch
    "val_acc":    0.763,                     # 驗證準確率
    "decision_threshold": 0.47,            # 自動掃出的最佳閾值
    "test_acc":   0.751,                     # 獨立 test set 結果
}

這種設計讓系統具備「自我描述」與「向後相容」能力 — 即使未來訓練不同架構或不同類別的模型,推論端不需要任何程式碼修改。

05 / DATA ENGINEERING 資料工程

深度學習有一個著名的公理:「Garbage in, garbage out.」— 資料品質決定模型上限。本章詳述我們的資料蒐集、人臉偵測創新(三層 Fallback)、資料增強與三段切分策略。

5.1資料蒐集

本研究的資料集由本組成員自行蒐集與標註,來源包括組員自身拍攝的照片、從影片中以 extract_frames.py 擷取的影格,以及部分公開素材。所有照片皆經人工分類為 Good(表情自然)或 Bad(表情崩壞)。最終資料集統計如下:

5.2核心創新:三層 Fallback 人臉偵測

在第二版原型中,我們僅使用 MediaPipe full-range model(model_selection=1),導致 27 張照片漏失。深入分析這些失敗案例後,我們將其歸類為三種模式:

  • 類型 A — 肖像照與自拍:full-range 模型針對遠距人臉優化,對「臉佔畫面 50% 以上」的近距大臉判斷不佳。
  • 類型 B — iPhone 螢幕截圖:iPhone Pro 系列橫向截圖尺寸為 2556×1179,但實際內容只在中間區域,兩側為純黑邊。MediaPipe 受黑邊干擾,無法定位人臉。
  • 類型 C — 極端 edge cases:手蓋住臉、極遠距人物等,任何單一模型都難以處理。

對策上,我們設計了三層 fallback 鏈:

def _detect_largest_face(bgr, min_face_px, margin):
    """三層 fallback,逐層降階嘗試。"""
    primary, fallback = _get_detectors()  # model 0, model 1

    # 第 1 層:short-range model(專長肖像)
    result = primary.process(bgr_to_rgb(bgr))
    if not result.detections:
        # 第 2 層:full-range model(專長合照中遠距人物)
        result = fallback.process(bgr_to_rgb(bgr))

    # 第 3 層:裁掉純色 letterbox 邊框後重試
    if not result.detections:
        cropped, (off_x, off_y) = _crop_letterbox(bgr)
        if cropped.shape != bgr.shape:
            result = primary.process(bgr_to_rgb(cropped))
            if not result.detections:
                result = fallback.process(bgr_to_rgb(cropped))

    if not result.detections:
        raise NoFaceDetectedError("三層偵測皆失敗")
    return _expand_bbox(result, margin, off_x, off_y)

Letterbox 偵測的核心邏輯:對影像每邊計算標準差,若 std < 8 表示是純色邊框,自動裁掉後再餵 MediaPipe。

5.4資料增強(Data Augmentation)

為了在小資料集條件下提升泛化能力,我們設計了多重資料增強策略,僅應用於訓練集(驗證/測試集保持原始狀態): 

train_tf = transforms.Compose([
    # 隨機裁切 + 縮放:模擬不同人臉框大小
    transforms.RandomResizedCrop(224, scale=(0.82, 1.0), ratio=(0.9, 1.1)),

    # 水平翻轉:模擬左右臉差異(50% 機率)
    transforms.RandomHorizontalFlip(p=0.5),

    # 隨機旋轉 ±15°:模擬手持拍照的傾斜
    transforms.RandomRotation(degrees=15),

    # 色彩擾動:模擬光線、白平衡、膚色差異
    transforms.ColorJitter(brightness=0.25, contrast=0.25,
                           saturation=0.2, hue=0.03),

    transforms.ToTensor(),
    transforms.Normalize(mean=IMAGENET_MEAN, std=IMAGENET_STD),

    # 隨機遮擋:模擬手、頭髮、配件遮住部分臉(20% 機率)
    transforms.RandomErasing(p=0.2, scale=(0.02, 0.1)),
])

每個增強策略都對應一種真實世界的變異:

  • RandomResizedCrop:使用者上傳的照片中人臉佔比可能差異很大,訓練時隨機裁切讓模型學會「以五官為中心」的特徵。
  • RandomRotation ±15°:手持拍照常有輕微傾斜。
  • ColorJitter:不同光線、相機、膚色下,色彩分布差異極大。
  • RandomErasing:遮擋是真實場景的常態(手指、口罩、太陽眼鏡),隨機塗黑一塊可顯著提升 robust。

5.5嚴謹的三段切分(Train / Val / Test)

傳統做法是將資料切為 train / val 兩段,但這在「調超參數時反覆觀察 val 表現」的情境下,可能導致 val acc 虛胖(實際上模型在不知不覺中過擬合到 val set)。為避免此陷阱,我們採用嚴格的三段切分:

切分比例用途是否影響訓練決策
Train70%實際更新權重是
Validation15%調超參、選最佳閾值是
Test15%最終評估,只看一次否

特別強調:test set 在訓練全程從未被看過、從未用於任何決策。我們僅在最終評估時,用獨立的 evaluate.py 腳本對它跑一次。這確保最終回報的 75.1% test accuracy 是真實的泛化能力。

研究嚴謹性 在前期實驗中,我們的「val 76.3%」其實是經過多次調參後的最佳分數,有可能已輕微過擬合到 val set。但「test 75.1%」是模型在完全沒見過的資料上的表現,這個數字才是真實能力的證據。

06 / TRAINING & TUNING 模型訓練與調優

本章詳述本研究的兩階段微調策略、正則化技術、推論優化,以及最關鍵的 V2 vs V3-Large 對照實驗。

6.1Two-Stage Fine-Tuning 策略

本研究採取「兩階段微調」策略,逐步釋放模型能力:

Stage 1 — 凍結 features,只訓練 classifier head

將 ImageNet 預訓的 backbone 全部凍結,僅訓練自定義的 2 類分類頭。此階段目的是讓 head 適配新任務,而不破壞預訓特徵:

def build_model():
    model = mobilenet_v3_large(weights=MobileNet_V3_Large_Weights.IMAGENET1K_V2)
    for p in model.features.parameters(): p.requires_grad = False
    model.classifier = nn.Sequential(nn.Dropout(0.2), nn.Linear(960, 2))
    return model

optimizer = optim.Adam(trainable, lr=1e-3, weight_decay=1e-5)
epochs = 15

Stage 2 — 解凍最後 3 個 inverted residual blocks

Stage-1 完成後,載入最佳 checkpoint,解凍最後 3 個 blocks(features[14:]),並用更小的學習率(1e-5,Stage-1 的 1/100)繼續訓練。這允許底層特徵也微幅調整,但避免大幅更新破壞預訓知識。

6.2正則化策略 — 對抗小資料集過擬合

Weight Decay 1e-4

L2 正則化,懲罰權重過大,迫使模型學習「簡單的解」而非「死背訓練資料」。

Label Smoothing 0.05

不讓模型對任何預測「100% 確定」,正確答案的 one-hot 從 1.0 平滑為 0.95,提升泛化。

Dropout 0.2

Classifier 加入 dropout,訓練時隨機關閉 20% 神經元,避免過度依賴特定特徵。

6.3推論優化(TTA + 閾值調校)

TTA — 水平翻轉版本平均

將每張測試照片同時餵入「原圖」與「水平翻轉版本」,將兩個預測機率平均,得到更穩定的最終預測:

def predict_image_quality(image_path):
    face = _detect_and_crop_face(image_path)
    batch = torch.stack([tf(face), tf(face[:, ::-1, :].copy())]).to(device)
    with torch.no_grad():
        probs = F.softmax(model(batch), dim=1).cpu().numpy()
    return probs.mean(axis=0)  # 兩版本機率平均

決策閾值自動掃描

傳統做法用 0.5 作為「Bad」與「Good」的分界,但這個切點在小資料集上未必最佳。我們在 val set 上掃描 0.30~0.70,找出真正讓 val accuracy 最高的閾值,並寫入 checkpoint。本研究最終掃出最佳閾值為 0.47。

8.4軟刪除與還原機制

「怕刪錯」是設計考量的核心痛點之一。我們設計了「軟刪除 + manifest 追蹤」機制,讓刪除動作完全可逆:

# backend/main.py — 軟刪除流程
@app.post("/api/trash")
def move_to_trash(req: TrashRequest):
    trash_dir = Path(req.folder) / "Trash"
    trash_dir.mkdir(exist_ok=True)
    manifest = []
    for path_str in req.paths:
        src = Path(path_str)
        dest = trash_dir / src.name
        if dest.exists():  # 避免覆蓋
            dest = trash_dir / f"{src.stem}_{timestamp}{src.suffix}"
        shutil.move(str(src), str(dest))
        manifest.append({
            "original_path": str(src),
            "trash_path":    str(dest),
            "deleted_at":    datetime.now().isoformat(),
        })
    # 追加到既有的 manifest.json
    _append_manifest(trash_dir / "manifest.json", manifest)
    return {"moved": len(manifest)}

選擇性還原

使用者可在 Trash 視圖中**勾選想還原的照片**,後端依 trash_paths 過濾;沒勾選的繼續留在 Trash: 

@app.post("/api/trash/restore")
def restore_from_trash(req: RestoreRequest):

            remaining.append(entry)
            continue
        shutil.move(entry["trash_path"], entry["original_path"])

09 / REFLECTION 反思與限制

沒有完美的系統。本章誠實面對本研究的限制,並反思開發過程中的關鍵領悟。我們相信,認識自己的邊界,才有可能突破它。

9.1學到的事

(1) 資料的價值大於模型

在前期實驗中,我們不斷嘗試「換更大的模型」、「加更多 epoch」、「調更花俏的超參數」,但 val accuracy 始終卡在 70-72%。讓我們突破到 75.1% 的關鍵,是把資料集從較少張擴增到 1,060 張,並更嚴格地檢視類別平衡。此經驗讓我們體會「Data-centric AI」 — 當模型已經夠好,投資在資料品質上的回報遠高於投資在模型架構上。

(2) 評估嚴謹性是真實能力的證據

引入嚴格的 train / val / test 三段切分後,雖然「report 出來的數字」變小(75.1%),但這個數字是真實的,我們對它有信心。研究的價值在於誠實,不在於漂亮數字。

(3) 失敗案例帶來最大的工程突破

本研究最關鍵的創新 — 三層 Fallback 偵測 — 並非來自高深的演算法,而是來自對失敗案例的詳細觀察。「為什麼模型抓不到這 27 張?」這個問題,引導我們發現 iPhone 截圖的黑邊現象,進而設計出輕量的 letterbox 偵測。

9.2系統限制

限制影響暫無解 / 未來方向
訓練資料規模小準確率天花板約 80%加更多資料是唯一根本解
族群多元性不足對訓練資料分布之外的人臉可能表現較弱擴增更多元的資料集
「廢片」主觀性高標註者間可能有意見不一致需要更明確的標註指南
極端 edge case 仍漏3 張電視截圖等仍無法偵測需要更強的 face detector
無連拍智選功能不能從 10 張連拍中選最佳需要 EXIF 整合 + 群組推薦演算法

9.3倫理考量

(1) 隱私保護

本系統完全本機運行,使用者的照片不會上傳任何雲端,這是設計原則而非妥協。對於含家庭、私密照片的使用者特別重要。我們也未在程式中加入任何分析、追蹤、上傳行為。

(2) 演算法偏見(Algorithmic Bias)

我們的訓練資料來源有限,模型對訓練資料之外的族群、場景、光線條件,可能有未驗證的偏差。我們在 README 與報告中明確揭露此限制,並建議使用者在多元場景下需謹慎驗證。

(3) 「廢片」定義權

什麼算「廢片」是高度個人化的 — 對某人「閉眼」可能是廢片,對另一人卻可能是珍貴的回憶。我們的系統設計為「AI 建議,使用者最終決定」 — 每張被歸到 Bad 的照片都需要使用者手動勾選才會被刪除,且軟刪除機制讓決定隨時可撤回。這保障了使用者的「數位自主權」。

CHAPTER 10

10 / CONCLUSION 結論與未來展望

本研究實作了一套完整的人臉表情品質自動評估系統,從資料蒐集、模型訓練、嚴謹評估到 Web UI 部署,體現「設計思考 × 人工智慧」的學習成果。

10.1研究貢獻總結

技術貢獻

  • 提出三層 fallback 人臉偵測架構,改善 88.9%
  • 完整 two-stage fine-tuning + 正則化 pipeline
  • 對照實驗驗證 SE Attention 對人臉任務的價值
  • 嚴謹 train / val / test 三段評估

產品貢獻

  • 端到端 Web UI,可立即使用
  • 軟刪除 + 還原機制,降低使用者焦慮
  • 本機運行,完全離線保護隱私
  • 透明的信心分布視覺化

學習貢獻

  • 從具體痛點出發,完成完整原型迭代
  • 展現「Data-centric AI」思維
  • 誠實面對限制與偏見
  • 跨領域(會計、統計、歐法、廣電)協作

10.2未來方向

  1. 資料擴增至 2,000+ 樣本:突破 80% accuracy 天花板的最直接方式。重點補足側臉、合照次要人物、多元族群。
  2. 連拍智選最佳功能:結合 EXIF 時間戳,自動將拍攝時間 ±2 秒的照片分組,在組內保留信心最高的 Good 照片。這是真正的差異化功能,既有工具都做不到。
  3. 桌面 App 打包:用 PyInstaller 或 Electron 打包成 .app / .exe,讓非技術使用者也能受益,無需在終端機執行指令。
  4. 使用者標註回饋機制:當模型判錯時,讓使用者點「實際上是 Good/Bad」,累積真實使用回饋資料,持續改進模型 — 這是 ML 產品的標準資料飛輪(Data Flywheel)。
  5. 多語言介面:目前 UI 為繁體中文,可擴增至英文、日文,觸及國際使用者。

10.3結語

從一個日常的痛點出發,到實作完整的 Python 程式碼、訓練自己的深度學習模型、設計完整 Web UI,本團隊 5 位來自會計、統計、歐法、廣電 4 個系所的同學,以跨領域協作的方式完成這個專案。

過程中我們深刻體會到:「AI 不是替代人類做決定的工具,而是讓人類更有餘裕做重要決定的工具」。我們希望這個系統,能讓使用者把節省下來的整理時間,用在真正享受相簿裡那些值得珍藏的瞬間。
