# Smart Album Cleaner Security Hardening Design

## Context

The application is intentionally a local photo-management tool, but its Windows setup instructions bind the FastAPI server to all interfaces. File-serving and trash routes accept absolute paths, the frontend toolchain contains known vulnerabilities, the legacy MediaPipe dependency forces an unsafe protobuf line, and model checkpoints are loaded with unrestricted pickle behavior.

The hardening must preserve folder selection, face detection, quality classification, previews, application Trash, restore, and system-trash workflows.

## Goals

- Make the web service loopback-only by default on every supported platform.
- Restrict file operations to roots explicitly selected for a scan during the current process.
- Migrate face detection from legacy `mp.solutions` to the supported MediaPipe Tasks API.
- Remove the vulnerable protobuf constraint and upgrade the frontend build toolchain.
- Load PyTorch checkpoints in weights-only mode and verify downloaded model assets.
- Preserve classification behavior and the current frontend API contract.

## Non-goals

- Changing the trained MobileNet classifier or retraining its weights.
- Uploading user photographs or scan metadata to a remote service.
- Adding multi-user accounts to a local desktop tool.
- Redesigning the Vue interface.

## Design

### Local service boundary

All documented launch commands will bind Uvicorn to `127.0.0.1`. The backend will accept browser origins only from the documented loopback development and production origins. Host/origin checks will reject non-loopback browser access before a file operation is evaluated.

### Authorized scan roots

The process will keep an in-memory registry of canonical roots selected through folder/file selection or accepted by `POST /api/scan`. Image reads, application Trash, restore, and system-trash operations will resolve symlinks and verify that every target remains within the associated canonical root. Traversal, sibling paths, mismatched manifests, and files outside the root will return a non-revealing 403/400 response.

This registry is intentionally ephemeral: restarting the local app requires selecting the folder again. No personal path allowlist will be committed.

### Face detection and dependencies

Legacy `mp.solutions.face_detection` will be replaced behind a focused detector adapter using `mp.tasks.vision.FaceDetector`. The adapter will expose the bounding-box contract currently consumed by preprocessing and prediction code, allowing the surrounding crop and classifier logic to remain unchanged.

The current MediaPipe Tasks package does not require the old protobuf dependency. The explicit `protobuf<4` pin will be removed, and the project will use an audited current MediaPipe release. The required face-detector TFLite asset will be downloaded from the documented upstream source or project release and verified against a committed SHA-256 value before use.

Vite and its Vue plugin will be upgraded together to audited compatible releases, with the lockfile regenerated by `npm install`.

### Safe checkpoint loading

Every project-controlled `torch.load` call will use `weights_only=True`. Existing checkpoints contain tensors and primitive metadata only; regression tests will create a representative checkpoint and verify prediction/evaluation loading. Documentation will instruct users to download only the project release asset whose checksum is listed.

### Errors

- Non-loopback or disallowed browser origin: 403 before path processing.
- File outside an authorized root or symlink escape: 403 without returning filesystem details.
- Missing detector/classifier model: the existing clear 503 or file-not-found guidance remains.
- Invalid or incompatible checkpoint: a clear local error, without falling back to unrestricted pickle loading.

## Testing strategy

Each change is delivered as a vertical TDD slice:

1. Write and fail a backend contract test for an image path outside an authorized scan root; implement canonical-root enforcement and make it pass.
2. Repeat for trash, restore, system-trash, and symlink escape boundaries one behavior at a time.
3. Write and fail an origin/host test; implement loopback-only request checks and update launch documentation.
4. Add adapter contract tests using synthetic detector results; migrate one face-detection consumer at a time while keeping crop outputs stable.
5. Add a representative checkpoint test that rejects unrestricted loading; switch each load site to `weights_only=True` one at a time.
6. Upgrade one dependency family at a time and run `pip-audit`, `npm audit`, Python tests, Vue build, and a real MediaPipe detector smoke test against a non-personal fixture image.
7. Run an end-to-end local API smoke test covering select root → scan fixture → preview → app Trash → restore.

## Portfolio documentation

The README will describe local-only processing, path confinement, safe checkpoint loading, model checksums, MediaPipe Tasks migration, and reproducible verification commands. It will explicitly state that photographs remain on-device.

## Acceptance criteria

- Documented server commands never bind to `0.0.0.0`.
- Browser requests from an unapproved origin cannot read or mutate local files.
- All image/trash targets are confined to an authorized canonical root.
- No project `torch.load` call uses `weights_only=False`.
- The legacy protobuf pin is gone and dependency audits report no known vulnerability in the resolved set.
- Existing prediction behavior, frontend build, and the end-to-end local photo workflow pass.
